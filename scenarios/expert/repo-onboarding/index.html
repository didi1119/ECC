陌生 Repo 接手模式

高手

先讓 ECC 自己畫地圖，再開始工作，不要你親自帶路

這頁解的痛點

很多 agent 一進陌生專案就開始亂改，真正缺的是先理解 repo。這個模式是先把探索、架構盤點、風險辨識交給 ECC 自己做完。

接手流程

1
先找入口
辨認 app 入口、主要資料流、關鍵設定與測試位置
2
畫出地圖
整理模組分工、依賴關係、最危險的改動點
3
列出風險
先說清楚哪裡容易踩到 auth、data、deployment
4
再提開工順序
確認先讀哪些檔、先驗證哪些假設、先動哪一層
5
最後才開始改
避免你還沒看懂 repo 就進場瞎修

接手陌生 Repo Prompt

很適合新 repo、舊專案、別人交接下來的 codebase
> 你現在接手一個陌生 repo，先不要急著實作。 請先自己完成這輪 onboarding： 1. 找出主要入口檔、核心模組、重要設定、測試位置 2. 用簡潔方式整理出架構地圖 3. 列出這個 repo 最可能出問題的 3-5 個風險點 4. 告訴我如果要開始改功能，合理的切入順序是什麼 除非遇到缺檔或需求不明，不要中途停下來問我。 你先把 repo 摸清楚，再進入下一步。

⚠️ 不要一開始就叫它『直接修』

對陌生 repo 直接進實作，通常會花更多時間在後面返工。先讓 ECC 產出地圖與風險，再開始改，反而更快。

最值得它先找的東西

1. 真正的入口頁面與主要路由。

2. 設定檔、環境變數、build 指令。

3. 關鍵資料模型或 API 邊界。

4. 測試在哪裡、目前 coverage 大概怎樣。

5. 哪些地方一改就可能牽連很多模組。

進一步搭配

從摸清專案直接接到執行
> 接手完 repo 後，下一步請進入不中斷交付模式，自己接管後續流程
