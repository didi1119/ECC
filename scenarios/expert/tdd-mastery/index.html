TDD 完整實戰

高手

RED → GREEN → IMPROVE 三階段完整教學，含常見陷阱

TDD 三階段

RED
先寫會失敗的測試
明確定義你要的行為，測試跑起來要是紅的
GREEN
最小實作通過測試
只寫讓測試通過的最少程式碼，不多不少
IMPROVE
重構不改行為
清理程式碼，全程保持測試是綠的

提示詞品質對比

提示越具體，Claude 產出的測試越精準：

/tdd 提示詞品質
❌ 模糊提示（Claude 會跳過 RED）
/tdd 做登入
→ Claude 產生：「我來幫你實作登入功能！」然後直接寫 LoginForm 元件...
（沒有先寫測試！）
✅ 具體提示（確保 RED → GREEN）
/tdd 實作 validateLoginForm(email, password) 驗證函式：
- email 格式不合法時回傳 { valid: false, error: "email 格式錯誤" }
- password 少於 8 字元時回傳 { valid: false, error: "密碼太短" }
- 兩者都合法時回傳 { valid: true }
→ Claude 先寫 3 個測試案例，跑 FAIL，再寫實作，跑 PASS ✅

🚨 Claude 有時會跳過 RED 直接寫實作

如果你的提示太模糊（例如「做登入」），Claude 可能直接產生元件而不先寫測試。解法：在提示裡明確寫出「先寫測試，確認 FAIL 後再實作」。

RED 階段示範

Claude 先寫測試，測試必須失敗才代表 RED 階段正確：

Claude Code
/tdd 實作 addToCart(item, quantity) 函式，應累加相同 item 的數量
U
C
先寫測試（RED 階段）： ```typescript // cart.test.ts it('應累加相同 item 的數量', () => { const cart = new Cart(); cart.addToCart({ id: 'a', price: 10 }, 1); cart.addToCart({ id: 'a', price: 10 }, 2); expect(cart.getItem('a')?.quantity).toBe(3); }); it('不同 item 應分開計算', () => { const cart = new Cart(); cart.addToCart({ id: 'a', price: 10 }, 1); cart.addToCart({ id: 'b', price: 20 }, 1); expect(cart.items).toHaveLength(2); }); ``` 現在跑測試 → 應該 FAIL（因為還沒實作）

⚠️ 一定要確認測試真的 FAIL

如果 Claude 寫完測試後跑起來直接通過，代表測試沒有真正驗證任何東西。要求 Claude「現在跑測試，給我看 FAIL 的輸出」。

GREEN + IMPROVE 階段示範

Claude Code
C
GREEN 階段 — 最小實作： ```typescript // cart.ts addToCart(item: Item, quantity: number) { const existing = this.items.find(i => i.id === item.id); if (existing) { existing.quantity += quantity; // 累加 } else { this.items.push({ ...item, quantity }); } } ``` 跑測試 → 應該全部 PASS ✅
測試通過了！接下來呢？
U
C
IMPROVE 階段 — 重構但不改行為： ```typescript // 提取輔助函式，加入型別，提高可讀性 private findItem(id: string): CartItem | undefined { return this.items.find(item => item.id === id); } addToCart(item: Item, quantity: number): void { const existing = this.findItem(item.id); if (existing) { existing.quantity += quantity; return; } this.items.push({ ...item, quantity }); } ``` 重構完再跑一次測試確認仍然全綠 ✅

💡 IMPROVE 階段只整理，不加功能

重構時只改程式碼的結構（提取函式、改命名、加型別），不加新邏輯。新邏輯 = 新的 RED 階段。

Coverage 怎麼看

跑 npx vitest run --coverage 後看報告：

FileStmtsBranchLines
cart.ts100%100%100%
checkout.ts72%60%71%

Branch coverage 低 = 有 if/else 分支沒被測試到。優先補這些 case，而不是追求 100% 行覆蓋率。

好的 /tdd 提示詞範本

❌ 太模糊，Claude 可能跳過 RED
> /tdd 做登入
✅ 具體明確，確保 RED → GREEN 流程
> /tdd 實作 JWT refresh token 邏輯，需求： 1. access token 過期（401）時自動用 refresh token 換新 2. 換新過程中的並發請求要 queue 起來，不能重複刷新 3. refresh token 本身過期時清除 session 並跳轉登入頁 4. 測試要涵蓋：正常刷新、並發刷新、refresh 過期三個 case
✅ 另一個好範例：明確指定邊界條件
> /tdd 實作 formatCurrency(amount, currency) 函式：TWD 用「NT$」前綴，USD 用「$」，超出範圍的 currency 要 throw Error
