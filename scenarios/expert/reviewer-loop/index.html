雙 Agent 評審迴圈

高手

把「做」和「審」拆成兩個角色，讓品質門檻自己執行

為什麼這招實用

你不用再自己扮演 PM、tech lead、reviewer 三種角色。先把評分規則講清楚，ECC 就能自己迴圈修正，而不是半成品就來問你下一步。

評審迴圈流程

1
指定雙角色
一個 Agent 只負責實作，一個 Agent 只負責評審
2
定義評分表
功能完整度、測試、可讀性、風險，各自有分數
3
設定門檻
總分未達 8/10 不得宣布完成
4
自動退件
reviewer 指出缺口後，developer 直接修正再送審
5
過線才結案
達標後再整理最終摘要與剩餘風險

可直接套用的工作流 Prompt

把 reviewer 變成真正的品質門神，而不是禮貌型附和 reviewer
> 請用雙 Agent 模式完成這個功能： - Agent A = developer，只負責實作 - Agent B = reviewer，只負責審查與打分 流程規則： 1. developer 完成一輪後，reviewer 依下列 rubric 打分： - 功能完整度 0-10 - 測試與驗證 0-10 - 可讀性與結構 0-10 - 風險控制 0-10 2. reviewer 要給出總評與明確缺失 3. 只要任一項低於 8 分，就退回 developer 修正 4. developer 修正後再次送 reviewer 5. 直到所有項目都 >= 8 分才可結束 6. 除非遇到架構方向或需求衝突，不要中途停下來問我

⚠️ 不要只寫『幫我 review』

如果沒有門檻、沒有 rubric、沒有退件條件，reviewer 很容易只給模糊建議，最後還是要你自己判斷到底能不能收工。

最適合用在

1. 功能要快速做完，但你又不想自己扛最後品質把關。

2. 多輪修正容易失焦，需要 reviewer 持續抓同一套標準。

3. 你想把「做到哪裡算完成」變成明確規則，而不是感覺。

快速改寫模板

換成你自己的品質 rubric
> 把 reviewer 的評分表改成『功能、測試、效能、安全、DX』五項，全部 >= 8 才能結案
減少重複退件與失焦
> reviewer 每輪都要列出『本輪新增發現』與『仍未解決項目』，避免來回鬼打牆
