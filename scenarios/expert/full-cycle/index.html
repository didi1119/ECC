把所有高手技巧串成一條完整的開發流水線

一個真實的功能需求，從規劃到 commit，走完 7 個步驟，串聯所有前面學過的技巧。

步驟 1

/plan — 規劃功能

套用提示工程技巧（背景 + 需求 + 非目標 + 格式要求）：

完整的 /plan 提示（可直接套用修改）
> /plan 為現有的 Next.js 14 + Supabase 專案新增「用戶大頭貼上傳」功能： 背景： - 現有 auth 系統使用 Supabase Auth（不要修改 auth 邏輯） - Storage：Supabase Storage 已設定好 bucket 需求： - 支援 JPG、PNG、WebP，最大 2MB - 上傳後自動裁切成 200x200 - 即時預覽，不用重新整理 - 舊大頭貼自動刪除（避免儲存空間浪費） 非目標： - 圖片濾鏡或編輯功能 - 大頭貼歷史紀錄 每個 phase 列出：受影響的檔案路徑、可獨立 merge

⚠️ 確認計畫才開始實作

收到計畫後一定要仔細檢查 phase 分割，確認 yes 之後才進入步驟 2。修改計畫比修改程式碼便宜 10 倍。
步驟 2

設定 Hooks（一次設定，長期受益）

在開始寫程式碼之前，先設定好自動化 hooks：

設定三道自動檢查關卡
> 幫我設定 PostToolUse hook：每次修改 .ts / .tsx 後自動跑 npm run format、npm run lint、npx tsc --noEmit

設定完成後，後面所有步驟的程式碼修改都會被自動驗證。詳見Hooks 設定教學。

步驟 3-4

/tdd — 核心邏輯 + API

先 TDD 核心驗證邏輯（最容易測試的部分）
> /tdd 實作 validateAvatarFile(file) 函式：驗證 MIME type（jpg/png/webp）、大小 ≤ 2MB、副檔名白名單，不符合時回傳明確錯誤訊息
再 TDD API 端點（整合測試）
> /tdd 實作 POST /api/avatar 上傳端點：驗證 → 壓縮到 200x200 → 上傳 Supabase Storage → 刪除舊圖 → 回傳新 URL

詳見 TDD 完整實戰教學。

🚨 測試全過才跑 /code-review

在測試還有紅燈時跑 code review 會浪費一整個 context 去分析不完整的程式碼。先確保綠燈，再 review。

Security Review — 檔案上傳安全審查： **CRITICAL** — 缺少檔案大小伺服器端驗證 ```typescript // 問題：只在前端驗證 2MB 限制，可被繞過 // 修復：在 API route 加入伺服器端驗證 if (file.size > 2 * 1024 * 1024) { return NextResponse.json({ error: '檔案超過 2MB' }, { status: 400 }); } ``` **HIGH** — 上傳路徑包含用戶可控的 filename ```typescript // 問題：path traversal 風險 const path = `avatars/${userId}/${file.name}`; // 修復：用 UUID 重新命名，完全不信任原始 filename const ext = getExtension(file.name); const path = `avatars/${userId}/${randomUUID()}${ext}`; ``` 建議：先修復 CRITICAL，再繼續其他工作

修復 CRITICAL 安全問題：

路徑遍歷漏洞修復
❌ 有路徑遍歷漏洞typescript
// 有安全漏洞的版本
const path = `avatars/${userId}/${file.name}`;
// 用戶可以上傳 "../../config/secret.env" 這樣的路徑
✅ UUID 重命名 + 白名單驗證typescript
// 修復後的版本
import { randomUUID } from 'crypto';

const ALLOWED_EXTS = ['.jpg', '.jpeg', '.png', '.webp'];
const ext = path.extname(file.name).toLowerCase();

if (!ALLOWED_EXTS.includes(ext)) {
  throw new Error('不支援的檔案格式');
}
if (file.size > 2 * 1024 * 1024) {
  throw new Error('檔案超過 2MB');
}

// 用 UUID 完全取代原始 filename
const safePath = `avatars/${userId}/${randomUUID()}${ext}`;

⚠️ 檔案上傳、Auth、支付 — 這三類必跑 /security-review

這三種功能有最多常見漏洞。不要跳過，否則上線後很容易出事。

💡 用 /plan 的輸出當 PR 描述

把規劃階段的計畫直接貼到 PR body，review 者能立刻理解每個 phase 的目標和風險評估。
步驟 7

Commit & PR

如果設定了 Stop hook，對話結束前會自動跑 npm run build 最終確認。

提交功能（Conventional Commits 格式）
> git add app/api/avatar app/components/AvatarUpload.tsx && git commit -m 'feat: add avatar upload with Supabase Storage'
