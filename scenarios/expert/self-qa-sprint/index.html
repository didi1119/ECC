自我驗收補漏衝刺

高手

讓 ECC 在結案前先自己挑毛病、自己補洞、自己再驗證一次

你真正想要的不是『完成』，而是『可交付』

很多任務卡在最後 10%，不是因為主功能沒做好，而是缺驗證、缺文件、缺邊界條件。這一招是把那 10% 變成自動收尾流程。

補漏衝刺流程

1
列出完成定義
功能、測試、review、文件、風險都要有明確標準
2
自己先找缺口
主動檢查沒測到的 case、沒跑的驗證、沒更新的文件
3
補完再驗證
把缺口補齊後重新跑 lint / test / build / review
4
整理殘餘風險
誠實列出還沒做與為什麼先不做
5
最後才宣稱完成
避免半成品過早結案

補漏 Prompt 模板

適合功能已經大致完成，但你不想自己一項項追問『有沒有測試？有沒有文件？』
> 在你宣稱完成之前，先執行一輪自我驗收補漏： 請自己檢查： 1. 功能需求是否全部覆蓋 2. 有沒有少掉的 edge case 或錯誤處理 3. 有沒有沒跑的 lint / test / build / type-check 4. 有沒有 reviewer 很可能會挑的結構或命名問題 5. 有沒有需要一起更新的文件、註解或使用說明 如果發現缺口，不要先回報我，先直接補上再重新驗證。 只有在你完成補漏後，再給我最終結果： - 你主動補了哪些東西 - 驗證結果 - 剩餘風險 - 後續可優化項目

最常見的自動補漏項目

1. 補 loading、empty、error state。

2. 補失敗路徑與 edge case 測試。

3. 補 README、使用方式或 migration 說明。

4. 補 lint、type-check、build 驗證。

5. 補 reviewer 一眼就會看到的命名、拆分、風險註記。

💡 要求它列出『主動補了哪些東西』

這一行很有用。因為它會迫使 ECC 不只是說「我檢查過了」，而是真的把補漏動作具體化，讓你一眼看出它有沒有認真做收尾。

進一步搭配

和雙 Agent 評審迴圈疊加
> 把這輪自我驗收結果交給 reviewer 再打一次分，低於 8 分就繼續補漏
把完成定義變成硬門檻
> 完成前必須通過 lint、test、build，否則不准宣稱完成
