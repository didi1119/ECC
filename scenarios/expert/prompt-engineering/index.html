高效 /plan 提示工程

高手

寫出讓 Claude 產出可執行計畫的提示詞

💡 為什麼提示詞品質這麼重要？

模糊的 /plan 會產出「做一個登入系統」這種沒有具體步驟的計畫，浪費大量 context window。 好的提示詞可以讓 Claude 在第一次就產出可直接執行的分段計畫。

有效 /plan 提示的四個要素

📍
背景（Context）
現有的技術棧、已完成的部分、不能動的地方
🎯
需求（Requirements）
具體的功能點，越精確越好
🚫
非目標（Non-goals）
明確說「這次不做」，防止範疇蔓延
📦
產出格式（Output format）
要求列出受影響的檔案路徑、風險等級、可獨立 merge 的 phase

提示詞對比

/plan 提示詞品質
❌ 模糊（產出無法執行的計畫）
/plan 做一個電商網站
✅ 具體（產出可立即執行的計畫）
/plan 為現有的 Next.js 14 專案（使用 Supabase + Stripe）新增「訂閱方案」功能：

背景：
- 現有 auth 系統已完成（不要動它）
- 資料庫：Supabase PostgreSQL
- 支付：Stripe（已有 API key 設定）

需求：
- 三種方案：Free / Pro($9.9/mo) / Team($29/mo)
- 升級/降級要立即生效
- 取消訂閱後當期結束才失效

非目標（這次不做）：
- 發票與收據功能
- 優惠碼系統
- 年繳方案

請分成可以獨立 merge 的 phase，每個 phase 列出：受影響的檔案路徑、風險等級

計畫產出品質對比

計畫品質差異
❌ 模糊提示的輸出
# 電商網站計畫

我會幫你做一個完整的電商網站，包含：
- 商品頁面
- 購物車
- 結帳流程
- 會員系統
...（20 行模糊描述，沒有具體步驟）
✅ 具體提示的輸出
# 訂閱方案功能計畫

## Phase 1：資料庫 Schema（低風險，可獨立 merge）
- 新增 subscriptions 表：id, user_id, plan, status, period_end
- 修改：supabase/migrations/003_subscriptions.sql

## Phase 2：Stripe Webhook 處理（中風險）
- 新增：app/api/webhooks/stripe/route.ts
- 處理：checkout.completed, customer.subscription.updated, customer.subscription.deleted

## Phase 3：前端方案選擇 UI（低風險）
- 新增：app/settings/billing/page.tsx
- 依賴：Phase 1 完成

風險：Phase 2 需要 Stripe test mode 驗證，建議先在測試環境跑完整 webhook 流程

🚨 沒有檔案路徑的計畫不可執行

計畫如果只說「建立一個 API」而不說具體路徑（如 app/api/stripe/route.ts），Claude 之後實作時每次都要重新猜位置，浪費 context。要求計畫必須包含完整路徑。

如何迭代計畫

計畫不滿意時直接在對話框說「調整」，不需要重新 /plan：

Claude Code
Phase 2 太大了，能分得更細嗎？
U
C
好的，Phase 2 拆成： **Phase 2a：Webhook 基礎設施** - 建立 webhook handler + signature 驗證 - 只處理 checkout.completed **Phase 2b：訂閱狀態同步** - 處理 subscription.updated / deleted - 更新 Supabase subscriptions 表 這樣每個 PR 更小，更容易 review ✅
Phase 1 要注意什麼 migration 的問題嗎？
U
C
Phase 1 風險點： 1. **現有 users 表** — 新增 subscriptions 表後需要 FK constraint，確認沒有孤兒資料 2. **RLS Policy** — subscriptions 需要設定 Row Level Security，否則用戶能看到他人訂閱 3. **回滾** — Migration 加上對應的 down migration 以備不時之需 建議：先在 Supabase Studio 手動測試 schema，確認後再寫進 migration file

💡 「可以獨立 merge」是最好的 phase 分割原則

每個 phase 完成後都應該可以 merge 到 main 而不破壞現有功能。這迫使 Claude 設計出更乾淨的介面邊界。

⚠️ 計畫確認後才開始實作

收到計畫後先仔細看，確認 phase 分割合理、風險評估正確再說 yes。一旦開始實作，修改計畫的代價很高。
